# OSHINAVI作業プラン

毎朝の作業開始時、必ずこのファイルを先に確認してください。

## 手順

1. ソート順（販売終了日が近い順）
2. サブエージェントで 4サイトクロスチェック
3. 結果を表でユーザーに報告
4. 修正案を提示 → 承認取得 → 適用
5. 次の1件へ

## ルール遵守（MEMORY.md 参照）

- **一次ソース(公式サイト)を優先的に探し、公式＋販売1サイトで照合するのが理想形** (2026-04-26追加)
- **楽天URLは必ずDeep Link形式に変換** (2026-04-26追加, OSHINAVI主目的=アフィリ収益)
  - 変換式: `https://click.linksynergy.com/deeplink?id=z9x6HLNpWco&mid=53531&murl=<URLencoded>`
- 2サイト以上で日付・会場が一致することを確認
- 楽天 > ぴあ > (ローチケ/e+) 楽天/ぴあがあればローチケ/e+は非表示
- 個別公演URLが必須
- ツアー・複数会場は1エントリ
- tickets[].date = 販売終了日
- 売り切れは soldout=true で別表示
- 期限切れは削除
- 価格は2サイト以上で一致した場合のみ登録
- 色分け：赤/オレンジ/緑/青/グレー

## 🔴🔴 明朝以降の最優先戦略 2026-04-27夜追加

### 大方針: 一気にダメエントリを掃除→残りを丁寧に判定
**「完全にダメなのはClaudeもわかる」→ 一括スクリーニングで大量削除 → 残ったエントリのみ慎重にクロスチェック**

### ① 一括スクリーニング（最優先・スクリプト自動実行）
verified=false 320件から「明らかにダメ」を自動抽出して一括削除提案：
- URL全null（lawson/pia/eplus/rakuten すべて null）
- 汎用カテゴリTOPのみ（例: `l-tike.com/concert/`, `t.pia.jp/`, `eplus.jp/sf/live/jpop` 等）
- `artist=name` で完全一致 かつ vague命名（例: 「○○ ライブ 2026」「○○ コンサート」）
- `dateLabel="2026年"` だけで具体性なし
- e+の `/sf/word/` キーワードURL（個別公演ではない）

→ スクリプトで一覧抽出 → ユーザー確認 → 一括削除  
→ **目標: 50-100件削減**（バッチ16〜18の傾向から推定）

### ② 重複エントリ検出（スクリプト自動実行）
全エントリから類似のものを検出：
- 同じ正式公演名で複数エントリ存在（id:1548 vs 1775のようなケース）
- 同じ artist + 同じ date
- e+のURL重複

→ 統合 or 削除提案。本日id:1775削除した類似ケースを撲滅

### ③ 残ったエントリをC案で並列クロスチェック
10件並列WebFetch/WebSearch → 1表で判定提案 → ユーザー一括承認方式（バッチ17の方式）  
1件ずつ判定確認は時間かかりすぎるので避ける

### ④ 保留タスク再試行
- id:1519 SPY×FAMILY展 名古屋 ローチケURL再試行
- id:977 BOYNEXTDOOR ローチケURL再試行

**新規イベントを index.html / events.html に追加する前に、必ず2サイト以上でクロスチェック完了させること。未検証のまま追加するのは絶対NG。**

### ⚠️ 「(仮)」扱いについて
**初期化時の verified=true 174件は「クロスチェック済み(仮)」扱い。**

- 過去のクロスチェック作業に明確な記録が残っていなかったため、
  「過去2日新規追加24件」と「販売終了日早い順 上位150件」を
  ユーザーの記憶ベースで verified=true として暫定的にマーク
- 厳密に2サイト一致確認した記録ではないため、信頼度は中
- **verified=false の508件すべてのクロスチェックが完了したら、
  この174件「(仮)」分も改めて全件クロスチェックする**

### 運用ルール
1. **クロスチェック完了**: 2サイト以上で日付・会場・アーティスト名一致確認 → `verified: true` を付与
2. **新規追加時**: 必ず `verified: true` で登録（クロスチェック完了が前提）
3. **データ修正時**: 内容修正後にクロスチェック完了したら verified=true 維持
4. **削除時**: そのエントリごと削除（フラグ操作不要）
5. **「(仮)」174件の再検証**: verified=false 508件すべて完了後にユーザーへ報告 → 再検証作業に移る

### 表示ロジック (index.html)
- `matchEvent()` の冒頭で `if (ev.verified !== true) return false;`
- `init()` で `EVENTS.filter(e => e.verified === true)`
- `allLabels` 生成時にも verified=true フィルタ

### 記録
- `tmp/verified_log.json` に初期化時のIDリストと件数を保存
- 今後の verified=true 付与作業はクロスチェック作業の中で行う
