# OSHINAVI作業プラン

毎朝の作業開始時、必ずこのファイルを先に確認してください。

## 手順

1. ソート順（販売終了日が近い順）
2. サブエージェントで 4サイトクロスチェック
3. 結果を表でユーザーに報告
4. 修正案を提示 → 承認取得 → 適用
5. 次の1件へ

## ルール遵守（MEMORY.md 参照）

- 2サイト以上で日付・会場が一致することを確認
- 楽天 > ぴあ > (ローチケ/e+) 楽天/ぴあがあればローチケ/e+は非表示
- 個別公演URLが必須
- ツアー・複数会場は1エントリ
- tickets[].date = 販売終了日
- 売り切れは soldout=true で別表示
- 期限切れは削除
- 価格は2サイト以上で一致した場合のみ登録
- 色分け：赤/オレンジ/緑/青/グレー

## 📋 作業の流れ（毎朝のルーチン）

### ① 期限切れチェック
- 期限切れを抽出→ユーザー確認→削除

### ② クロスチェック継続
- ソート順の上から未検証のイベントを1件ずつ

### ③ 新着取得
- e+ や 楽天から新着、4サイトクロスチェックで登録

### ④ 1日2回までの push（承認必須）

## 🔴🔴 最重要ルール: 新規追加前のクロスチェック必須 2026-04-25追加

**新規イベントを index.html / events.html に追加する前に、必ず2サイト以上でクロスチェック完了させること。未検証のまま追加するのは絶対NG。**

- 楽天/ぴあ/ローチケ/e+ のうち2サイト以上で **日付・会場・アーティスト名** が一致することを確認
- 一致しない、または1サイトしか見つからない場合は追加保留 → ユーザーに相談
- 既存エントリのクロスチェック作業中に「新着取得→未検証で追加」をやってしまうと、後から検証作業が増えて二度手間になる
- 朝のルーチン③「新着取得」も同じルール: 新着発見 → 2サイト一致確認 → 登録

**例外なし。「とりあえず追加して後で検証」は禁止。**

**【絶対原則】クロスチェック済みのものしかOSHINAVIに載せない。**

未検証エントリ（verified=false）は何があっても表示しない。承認待ち期間中も同様。

### ⚠️ 「(仮)」扱いについて
**初期化時の verified=true 174件は「クロスチェック済み(仮)」扱い。**

- 過去のクロスチェック作業に明確な記録が残っていなかったため、
  「過去2日新規追加24件」と「販売終了日早い順 上位150件」を
  ユーザーの記憶ベースで verified=true として暫定的にマーク
- 厳密に2サイト一致確認した記録ではないため、信頼度は中
- **verified=false の508件すべてのクロスチェックが完了したら、
  この174件「(仮)」分も改めて全件クロスチェックする**

### 運用ルール
1. **クロスチェック完了**: 2サイト以上で日付・会場・アーティスト名一致確認 → `verified: true` を付与
2. **新規追加時**: 必ず `verified: true` で登録（クロスチェック完了が前提）
3. **データ修正時**: 内容修正後にクロスチェック完了したら verified=true 維持
4. **削除時**: そのエントリごと削除（フラグ操作不要）
5. **「(仮)」174件の再検証**: verified=false 508件すべて完了後にユーザーへ報告 → 再検証作業に移る

### 表示ロジック (index.html)
- `matchEvent()` の冒頭で `if (ev.verified !== true) return false;`
- `init()` で `EVENTS.filter(e => e.verified === true)`
- `allLabels` 生成時にも verified=true フィルタ

### 記録
- `tmp/verified_log.json` に初期化時のIDリストと件数を保存
- 今後の verified=true 付与作業はクロスチェック作業の中で行う
