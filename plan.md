## ⚡ 朝の期限切れチェック手順（必ずこのスクリプトを使う）

```bash
python tools/check_expired.py
```

- このスクリプトは **events.html の `endDate`** と **index.html の `tickets[].date` 全件** を見て判定する
- **絶対に自前で抽出スクリプトをゼロから書き直さない**（2026-05-17に endDate 見落としで139件誤判定の事故あり → feedback_check_existing_logic.md）
- 判定ロジックを変える時は `tools/check_expired.py` を直接編集し、events.html の `getStatus()` と一致させる
- **出力は「期限切れ削除」と「⚠️要再確認(販売終了日不明)」に分かれる（2026-05-21〜）**
  - 「期限切れ削除」= **削除前に必ずWebFetchで実態確認**（2026-05-29追加 → feedback_pre_delete_webfetch_verify.md）。販売中チケットが残ってたらticket.date更新で復元、全て販売終了なら削除実行。
  - **「⚠️要再確認」= `saleEndUnknown:true` のもの。絶対に勝手に削除しない。** 表示される売り場URLをWebFetchして本当の販売終了日を確認→`tickets[].date`を書き換える（買えなくなっていれば削除、まだ不明なら翌日再確認）。これは「終了日不明のチケットを発売日翌日に誤削除しない」ための仕組み → feedback_unknown_end_date.md

### 期限切れ「削除」リストへの実態確認手順（2026-05-29追加）

ticket.date は新着登録時の販売終了日のまま固定されることが多く、実際は公演直前まで延長されているケース多発。check_expired.py は WebFetch しないので、機械判定だけ信じると販売中エントリを誤削除する。

事故例（2026-05-29朝）:
- id:238 前橋ウィッチーズ: data 5/28 → 実態 5/30 23:59
- id:242 LEE YOUNGJI 東京: data 5/27 → 実態 6/4 23:59
- id:207 赤坂檜町テキサスハウス: data 5/28 → 実態 5/31 12:30（当日引換券に切替）

手順:
1. 「期限切れ削除」リストの全エントリの `links.pia / eplus / lawson / rakuten` のうち主要URLを並列WebFetch
2. 「受付中」「販売中」が残ってる → ticket.date を実態に合わせて更新（**削除しない・復元**）
3. 全て「販売終了」「受付終了」 → 削除実行（ユーザー「削除」承認後・再確認不要）
